# kk-doc-lifecycle 工作流

## 1. 先选模式

- 用户要“看报告 / 先判断”：
  走 audit-only，只输出分类和建议动作
- 用户要“直接处理 / 直接归档 / 帮我收拾”：
  走 execute，直接落高置信度更新或归档
- 用户没有特别说只看报告，但明确是在做项目空间文档治理：
  默认也走 execute，并持续完成后续高置信度阶段

## 2. 再选范围

- 单任务收口：
  只检查 active boundary 涉及的文档与入口
- 单目录治理：
  只检查用户点名的目录，例如 `archive/baseline/example-baseline/docs/`
- 项目空间治理：
  先按入口层分批检查，不要一口气无边界扫全仓

## 3. 每份文档的分类顺序

1. 它是不是当前真源或当前默认入口
2. 如果是，它是否需要同步更新
3. 如果不是，是否需要提炼有效部分迁入新真源
4. 如果只剩历史价值，是否可归档
5. 如果原文结构已经不可信，是否应 rewrite 再归档旧文

## 4. 默认落地顺序

1. 补当前真源
2. 更新索引或跳转说明
3. 再归档旧文
4. 必要时补 decision / learning

如果范围是“项目空间治理”，默认按三段式连续执行：

1. Phase 1:
  更新入口、说明、当前真源引用
2. Phase 2:
  归档已退出主线的兼容页、备份稿、旧入口页，并修正当前正文中的过时定位说明
3. Phase 3:
  清理杂项噪音，复核是否还存在高置信度归档候选

只有在以下情况才中断并等待用户：

- 归档门禁不充分
- 新发现的候选对象存在明显争议
- 范围突然扩到未声明边界的大目录

## 5. 何时回退到其他 skill

- 需要先补任务边界：
  回退到 `kk-task-kickoff`
- 已明确是实现收口、review、QA 或 spec 同步：
  优先用 `kk-doc-sync`
