# kk-doc-lifecycle 归档门禁

## 必须同时满足

1. 已确认文档不再属于当前真源主链
2. 已确认替代它的当前真源或入口位置
3. 当前团队仍能从新入口完成协作，不依赖旧文继续实现
4. 归档后不会让用户误以为“信息消失了”

## 优先归档的常见类型

- 旧说明、旧原型、旧协作文档
- 已被新版 `stack/specs/`、`.gstack/` 真源或 `archive/baseline/` 归档说明替代的旧文
- 只保留历史追溯价值的阶段性草稿

## 不要直接归档的情况

- 当前仍被入口文档指向
- 当前仍是唯一可解释某个行为、接口或流程的文档
- 只是内容没更新，但本质上仍应继续作为真源
- 尚未决定该迁到 `stack / .gstack / archive` 哪一层

## 归档后的最小收尾

- 更新旧入口的索引、跳转页或引用
- 必要时在新真源或 decision 中注明替代关系
- 若这是长期可复用判例，回写 `doc-placement.md` 或 `.gstack/decisions/`

## 默认执行节奏

对项目空间文档治理，优先按这个顺序自动推进：

1. 先改入口和引用
2. 再归档兼容页、备份稿、旧入口页
3. 最后清理 `.DS_Store` 等环境噪音并复核

不要在入口仍指向旧文时抢先归档；也不要在已经完成高置信度阶段后无故停住，要求用户重复输入“继续”。
