# KK Dev Skeleton 伙伴快速入口

这份只给普通伙伴看。你不需要理解 adapter、boundary、gate、evidence 这些内部词，直接在 Codex 里说清目标即可。

## 第一次安装插件

在 Codex 新线程里说：

```text
请帮我安装 KK Dev Skeleton 的 Codex plugin。

Marketplace Git 地址：https://github.com/duanxibo/kk-ai-dev-skeleton.git
Marketplace name：kk-dev-skeleton-internal
Plugin name：kk-dev-skeleton-adoption

安装完成后告诉我下一步怎么把当前项目接入。
不要连接真实数据、生产环境、数据库，也不要执行 git commit、push 或部署。
```

## 把当前项目接入骨架

安装后新开一个 Codex 线程，在目标项目里说：

```text
请使用 KK Dev Skeleton Adoption，把当前项目接入 KK Dev Skeleton。

项目名：<项目名>
主要用户：<谁会用这个项目>
项目目标：<这个项目要解决什么问题>
源码位置：不知道请你判断
文档位置：不知道请你判断
本次不要碰：真实数据、生产环境、数据库、代码提交流程，以及我明确说不能改的目录。

请你完成接入自检，告诉我已经完成什么、还缺什么、建议第一个低风险试点任务是什么。
```

## 后续正常开发

接入完成后，你可以继续用自然语言说需求：

```text
请帮我实现 <功能目标>。
用户应该能 <用户行为>。
完成后我希望看到 <可见结果>。
本次不要碰 <目录 / 模块 / 风险动作>。
```

Codex 会自己处理内部流程，并在需要你确认业务口径、真实数据、生产、数据库、破坏性操作或代码提交流程时停下来问你。

## 怎么判断接入成功

你只需要看 Codex 的总结里有没有这几项：

- 已创建或更新项目规则。
- 已完成本地自检。
- 已说明哪些信息还缺。
- 已给出第一个低风险试点任务。
- 已明确没有默认触碰真实数据、生产、数据库或代码提交流程。

## 更新插件后改造已有项目

如果插件提示有更新，对 Codex 说：

```text
请升级 KK Dev Skeleton 插件，并按最新版骨架升级当前项目。
先生成增量升级计划，不要重新创建项目，不要覆盖现有项目规则，不要移动业务代码。
```

更新骨架不等于重新做项目。Codex 应优先给出增量计划，只做安全、可回退的补齐。
